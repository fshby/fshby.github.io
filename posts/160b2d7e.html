网络接口基础

OSI 参考模型

  1. OSI七层模型
    1) 应用层
    2) 表示层
    3) 会话层
    4) 传输层
    5) 网络层
    6) 数据链路层
    7) 物理层
  2. TCP/IP四层模型：
    1) 应用层(HTTP/FTP/SMTP/Telnet)
    2) 传输层(TCP/UDP)
    3) 网络层( ICMP、IP、IGMP)
    4) 链路层(ARP、RARP)

    网络互连

  3. 集线器 —广播式
  4. MAC 地址 — 设备唯一的物理地址编号
  5. 交换机与以太网 — 不断地记录MAC地址，形成自己物理映射表，由广播式转为一对一交换信息，可以记录地址，
  6. 路由器 — 记录映射地址，不同的端口有不同的MAC地址
  7. 局域网内信息发送 信息发送时会包含原地址与目标地址信息：数据链路头部（源/目标IP地址）+网络层头部（源/目标物理地址）经过路由器，要由路由器转发（根据内部MAC表转发消息）
  8. 子网与子网掩码 —判断是否为同一局域网端口（IP地址与子网掩码同位与，第三个号段为端口号）
  9. 默认网关 （路由器） —对局域网外部设备发送消息，由理由器进行消息转发，
  10. arp缓存表与arp协议(地址转发协议/地址映射协议)—记录IP对应的MAC地址知道互相 的IP地址通过子网掩码判断统一子网否
  11. 下一跳 —路由器之间的跳转

封装与分用

TCP协议头部
SYN序列号 —对传输数据排序，每个数据的序列号+1，确保数据的正确性
ACK确认序号 — 数据过大分包发送时，确认分包序号连续性，保证数据分包不缺失
丢包—丢包之后期望序号不变，客户端继续从期望序号开始补发送包

建立连接与释放连接

建立连接三次握手

  1. 第一次握手：关闭状态的客户端发送链接请求同时进入同步已发送状态，监听状态的服务器接收到链接请求信息
  2. 第二次握手：服务器监听到客户端申请后发送连接请求确认同时进入同步已接收状态，客户端接收到链接请求确认信息
  3. 第三次握手：客户端发送确认信息同时进入链接建立状态，服务端接收到确认信息同时也进入链接确认状态
    建立连接，三次握手

释放连接

  1. 第一次挥手：客户端发送链接释放请求同时进入终止等待，服务端接收链接释放请求同时进入关闭等待
  2. 第二次挥手：服务端发送确认信息，客户端接收确认信息进入终止等待2
  3. 第三次挥手：服务端链发送完剩余数据后发送链接释放请求，客户端接收链接释放请求同时进入时间等待状态
  4. 第四次挥手：客户端发送确认信息，服务端接收确认信息同时进入关闭，客户端等待时间结束后自动关闭
    释放连接，四次握手
  • 注意：
    FIN创建关闭连接 —1为断开连接，

接口类型：

HTTP接口
Web Sercer接口
RESTful接口

HTTP请求组成

  1. 请求行
    1) 请求方式 —GET(查询动作)、POST（新建资源）、PUT(更新资源)、DELETE(删除资源)
    2) URL
    3) 协议版本
  2. 请求头 —键值对组成
    存放客户端的信息，Content-Type（请求数据类型）、token（用户身份令牌，token名字不唯一）
    1) Content-Type（请求数据类型）：
    application/json —json类型
    application/x-www-form-urlencoded：form表单数据被编码为
  3. 请求体 — 存放数据参数，POST与PUT请求的参数数据在请求体中，GET请求的参数在URL?后

HTTP响应组成

  1. 状态行
    1) 状态码
    1××—指示信息、表示请求已接收
    2××—表示请求已被成功接收，但是不代业务处理（内部代码问题）成功、
    3×× —重定向，先要完成某个动作，才能继续进行当下请求、
    4×× —客户端错误、
    5××—服务器端错误
    2) 协议版本号
    3) 状态消息

  2. 响应头
    1) 用于描述服务器的基本信息，以及数据描述
    2) Set Cookies：设置客户端本地cookies缓存

  3. 响应体
    响应的消息体，数据可以是普通文本、XML、JSON、HTML源码

HTTPS 和 HTTP 的区别主要如下：

Https：是以安全为目标的Http通道，是Http的安全版。Https的安全基础是SSL。

  1. http 是超文本传输协议，信息是明文传输，https 则是具有安全性的 ssl 加
    密传输协议。所以 HTTPS 协议更安全
  2. HTTP 协议进行数据传输比 HTTPS 效率高。
  3. http 和 https 使用的是完全不同的连接方式，默认端口也不一样，前者是
    80，后者是 443
  4. 因为 HTTPS 协议更安全但传输效率低，所以一般会用在对安全性要求比较
    高的系统中，例如网银、支付宝、商城系统系统。不过因为现在网络效率很
    高，所以 HTTPS 的传输效率低可以忽略。其实 app 的网络传输用的也是
    https 协议。

HTTP 状态码

HTTP 状态码 2xx,3xx,4xx,5xx 分别是什么意思？这个是最基本的了，这个得
熟练掌握，如果这个状态码都分不清，基本功就很弱了，印象分会大打折扣！

  1. 200请求已成功，请求所希望的响应头或数据体将随此响应返回。
  2. 403 服务器已经理解请求，但是拒绝执行它。与 401 响应不同的是，身份验证并不能提供任何帮助，而且这个请求也不应该被重复提交
  3. 404 请求失败，请求所希望得到的资源未被在服务器上发现
  4. 500 服务器遇到了一个未曾预料的状况，导致了它无法完成对请求的处理。一般来说，这个问题都会在服务器的程序码出错时出现。
  5. 501 服务器不支持当前请求所需要的某个功能。当服务器无法识别请求的方法，并且无法支持其对任何资源的请求。
  6. 502 作为网关或者代理工作的服务器尝试执行请求时，从上游服务器接收到无效的响应。
  7. 503 由于临时的服务器维护或者过载，服务器当前无法处理请求。这个状况是临时的，并且将在一段时间以后恢复。

接口测试：

接口测试原理 — 用工具模拟客户端向服务器发送请求，服务器处理相应业务，将信息反反馈给客户机，

为什么要开展接口测试

  1. 接口测试属于集成测试，测试介入越早越好，越在项目早期发现问题，其修复问题
    的成本越低
  2. 接口测试非常快速，UI 自动化执行一个测试用例 10S，接口测试用例执行话，需
    要时间是毫秒级
  3. 因为接口测试可以自动化

接口用例设计思路：

单接口测试： —验证每一个入参

  1. 正向功能：
    1) 全部参数
    2) 仅必填参数
    3) 部分参数组合
  2. 反向功能：
    1) 参数异常 —无参，少参，多参，参数拼写错误
    2) 数据异常 — 数据为空、数据长度不符、数据类型不符、错误数据错误数据
    3) 业务数据异常 —结合业务功能考虑输出的各种异常情况，跨业务调用接口

    多接口组合测试：

    接口之间的数据依赖，上下游接口依赖关系
    c宠物医院相关接口

    接口作业评讲：

    错误参数 —参数名称拼写错误
    请求方式 —GET请求一定是form表单格式
    Content-Type请求数据类型 — form类型与json类型
    请求参数 —form表单格式不加{ }，json内容类型无参是也需要加{}
    多参，注意上下接口关联性 —
    上一个接口的出参是下一个接口的入参

fiddler,HTTP协议调试代理工具

功能

  1. 抓包
  2. 接口拦截
  3. 伪造客户请求，伪造服务器响应
  4. 查看HTTP细节
  5. shift+R 较小压力测试，重复请求接口
  6. 弱网测试：
    1) 搜索更改内容 1) 更改时间后生效—Rules→performance→Simulate Modem
    SpeedsRules-performance：规则表现(可能是某种评价指标的术语，需要根据上下文进一步理解)
    Simulate Modem Speeds：模拟调制解调器传输速度
  7. fiddler工具常用命令
    1) ？text —-URL
    2) > — body
    3) =200 — result
    4) @host — Host
    5) clear —清空
    6) bold —字体加粗
    7) bpafter —组织

遇到问题

使用fiddler 对雷电模拟器进行手机端抓包时报错

  1. 报错内容：模拟器端浏览器无法访问网址 —本地IP:端口号
  2. 解决办法：更换fiddler端口 —-此处将8888更换为8887
  3. 注意事项：
    1) 要将设置手机端代理，手动配置为同一局域网下的fiddler工具电脑IP地址，端口为fiddler设置端口
    2) 安卓7版本以上，证书对用户不信任，不能进行抓包
    3) 雷电模拟器3版本一下，可以抓包
